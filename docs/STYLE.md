# Documentation style

This repository follows the Google developer documentation style guide for structure, tone, and release communication.

## Core rules

- Write in plain English.
- Start with the user task, not background history.
- Use short sections with descriptive headings.
- Prefer active voice and imperative steps.
- Put prerequisites before commands.
- Keep notes short and only use them when they change user action.
- Use lists only when the content is naturally list-shaped.

## Repository-specific rules

- State whether a feature is iOS-only.
- State the pinned `lynx` commit when behavior depends on upstream.
- Avoid promises about Android or Harmony parity.
- Include exact commands for setup, build, and release.
- Keep release notes focused on user-visible changes, fixes, and known limits.
